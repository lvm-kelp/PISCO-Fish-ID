# Sea Life Flashcards — Context for Claude

## What this app is

A static, single-page web app that helps the user (Greg) memorize the names of sea creatures via flashcards. Each card has an image on one side and the species name on the other. The user reveals the name, then marks the card as **"I know it"** (removed from the rotation) or **"Practice"** (stays in the rotation). The deck is randomized. There is a one-step **Undo** for accidental taps and a **Reset** to put all cards back.

- **No login. No backend. No long-term server storage.**
- Per-user "known" state persists across page reloads via `localStorage` only.
- Mobile-first; desktop is just a centered narrower layout.
- Target deck size: ~100 cards (final set will be sea life; current `images/` has placeholder photos — `banana.jpg`, `greg.png`, `mona.jpg` — that the user will replace).

## Tech stack and why

- **React 18 + Vite + TypeScript** — fast dev, trivial static build, deployable to Vercel/Netlify/GitHub Pages.
- **Tailwind CSS v3** — mobile-first styling without writing component CSS.
- **No router, no state library.** Single-page, single-component-tree app. Plain `useState`/`useEffect` is sufficient and avoids unnecessary abstraction.
- **No test framework yet.** Manual testing is fine for this size; if logic grows (e.g., spaced-repetition weighting), add Vitest around `pickRandom` and the filename parser.

## Key design decisions (made deliberately — push back before changing)

1. **Two-stage image pipeline.**
   - **`images/`** is the user-managed source of truth. Raw photos live in **per-category subfolders** (`images/fish/`, `images/algae/`, etc.) — any of `.jpg/.jpeg/.png/.webp`, any case, any naming. **Files at the `images/` root are skipped with a warning** — move them into a category folder to include them.
   - The set of valid category subfolders is defined by the `Category` type and `categoryToFolder` map in `src/types.ts` (`Fish`/`fish`, `Algae`/`algae`, `Inverts`/`inverts`, `UPC`/`upc`). Add a category by extending both, then creating the matching subfolder.
   - **`scripts/process-images.mjs`** (run via `npm run process-images`, also wired as `prebuild` so `npm run build` runs it automatically) walks each `images/<category>/` subfolder, cleans each filename (strips embedded `jpg|jpeg|png|webp`, splits on `-_` and whitespace, title-cases each word, rejoins with spaces), compresses with `sharp` (resize: max 1200px on longest edge, JPEG quality 82, EXIF rotation honored), and writes to **`images-processed/<category>/`** as `.jpg`. The script is incremental (skips when dest is newer than source), removes orphan files **per category**, and removes orphan category folders entirely.
   - **`cards.ts`** globs `/images-processed/**/*.jpg` via `import.meta.glob` — recursive across category folders, *not* the raw `images/` folder. Adding a photo to `images/<cat>/` doesn't show up until you re-run `process-images` (or run `build`, which does it via the `prebuild` hook). The path segment before the filename is parsed to determine the card's `Category` via `folderToCategory` — unknown folders are silently dropped, so renaming a subfolder without updating `types.ts` makes its cards disappear.
   - **Name collisions are fatal — per category.** If two source files in the *same* category clean to the same target name, the script exits with an error. Same filename across *different* categories is fine (each lives in a separate processed subfolder).
   - At ~150 sea-life photos (Fish only): raw `images/fish/` was 31MB, processed `images-processed/fish/` is 4.6MB (~85% reduction).
   - **Case-insensitive filesystems are handled** (macOS APFS default), per category. The script reads each `images-processed/<category>/` upfront and builds a lowercased lookup map. If a target's case differs from the on-disk name (e.g. user changed cleaner logic and `Oyt Foo.jpg` should now be `OYT Foo.jpg`), the script unlinks the old-cased file before sharp writes the new one — so renames take effect in a single pass. Output line `[fish] ↻ recasing OLD → NEW` shows when this happens. Don't remove this dance; without it, sharp's write silently keeps the old casing on case-insensitive FS.
   - **Acronyms are preserved**: a word matching `/^[A-Z][A-Z0-9]+$/` (uppercase letter followed by 1+ uppercase letters or digits) keeps its case. So `OYT olive-yellowtail.jpg` → `OYT Olive Yellowtail.jpg`, and `KGBC kelp-gopher.jpg` → `KGBC Kelp Gopher.jpg`. Same logic exists in both `process-images.mjs` (`titleCaseWord`) and `cards.ts` (`titleCaseWord`) — keep them in sync if you change the rule.
2. **Filename → display name** (in `cards.ts`):
   - Strip extension.
   - Replace `-` and `_` with spaces (mostly a no-op now since `process-images` already converted them).
   - **Strip trailing digits** (`\s*\d+$`) — multiple photos of the same species in `images/` are named `Black Rockfish.jpg`, `Black Rockfish 2.jpg`, `Black Rockfish 3.jpg`. The disk filenames stay numbered (uniqueness), but the **display name** drops the suffix so all three read "Black Rockfish" on the card back. Each is still a separate card with its own ID, so marking one known doesn't hide the others (intentional — different photos test recognition robustness).
   - Title-case each word (mostly idempotent on processed input).
   - The parser still works on raw filenames if a developer ever points it back at `images/`.
   - No sidecar `names.json` override exists yet — if names get awkward (apostrophes, italics, scientific names), discuss with user before adding one.
3. **Card `id` is `"<category-folder>/<filename>"`** (e.g. `fish/Black Rockfish.jpg`), not a UUID. Stable across builds, used as the localStorage key entry. The category prefix gives per-category independence of "known" state for free — a single flat `Set<string>` works because IDs from different categories never collide. **If you change ID derivation again, bump the storage key from `flashcards:v2:known` to `v3`** — old entries won't match and silently dropping known status is acceptable.
4. **Practice cards are deferred to the back of the rotation** (one-deep Leitner-style). Clicking "Practice" adds the card's id to `deferredIds: Set<string>`. The picker (`pickNextCard` in `pickRandom.ts`) first tries to pick from the *fresh pool* (`active \ deferred`); only when fresh is empty does it pick from `active` (which is then all-deferred). Order within deferred is **uniform random** — the user opted for this explicitly over FIFO when this feature was added. **No weighting / spaced-repetition beyond this** — don't introduce ease factors, intervals, or boxes without asking. `deferredIds` is persisted to `localStorage` under `flashcards:v2:deferred`, scoped per-category by ID prefix (same pattern as `knownIds`).
5. **Avoid same-card-twice-in-a-row** in `pickRandom.ts`. `pickRandomCard` rerolls once if `avoidId` matches and there are >1 candidates. The "avoid" memory is **one card deep** — after one intervening card, anything is fair game. Don't extend to a multi-card history without asking.
6. **Image fit is `object-fit: contain`** (switched from `cover` because cropping was hiding parts of the fish — identifying features like fin shape and tail markings matter for ID). Letterbox space on the card front is plain white. Don't switch back to `cover` without asking.
7. **Card aspect ratio is `aspect-[8/5]`** (≈ 1.6 landscape) — chosen as the median aspect ratio across all 150 processed images. Verified with sharp metadata: 146 of 150 photos are landscape, median ratio 1.595, mean 1.661. A landscape card with `contain` minimizes letterboxing. Don't revert to portrait (`aspect-[3/4]` etc.) without re-running the stats.
8. **Flip animation is CSS 3D transforms**, ~700ms `rotateY`. Implemented with hand-rolled utility classes in `index.css` (`perspective-1000`, `preserve-3d`, `backface-hidden`, `rotate-y-180`) — no Tailwind plugin dependency.
9. **Undo is one step only** (last action). Stored as `LastAction = { prevKnown, prevDeferred, prevCurrentId, prevIsFlipped } | null` — a **full pre-action state snapshot**, not a diff. Each Known/Practice handler calls `snapshot()` before mutating, and `handleUndo` blindly restores all four fields. The snapshot stores Set *references* (not copies); the helpers `withAdd` / `withDel` produce new Sets only when something actually changed, so the pre-mutation Set ref stays valid as the snapshot. **Why snapshot-based instead of diff-based**: the reverse logic stays uniform regardless of which sets a forward action touches — if a future action adds another piece of state, just include it in the snapshot rather than threading a new "wasInX" flag through both forward and reverse paths. `handleUndo` also clears any pending `flipResetTimer` so an in-flight card-swap can't fire after the undo restores state. Don't expand to a stack unless asked.
10. **Mid-flip card swap** (in `advanceToNext`): when the user marks a card, `setIsFlipped(false)` runs immediately, and the new card's `currentId` is set on a `setTimeout(..., CARD_SWAP_DELAY_MS)`. **Currently `CARD_SWAP_DELAY_MS = 0` in `App.tsx` — the user toggled this themselves** (see commits f05e12f / 27860b3 / 9ebca06 oscillating between 0, 350, and 700), and the current 0 means the new card's image becomes visible while the previous back-face is still rotating away. The original design called for ~50% of the 700ms flip duration (i.e. 350ms), which masks the swap behind `backface-visibility: hidden` once the rotation passes 90°. If you change `CARD_SWAP_DELAY_MS`, the trade-off is: low values = snappier, but the next card's image flashes during the unflip; ~350ms = visually clean but feels slower. Don't fire the swap synchronously without setTimeout — at minimum the React state update needs to land as a separate macrotask.
11. **Reset has a `window.confirm` prompt**. The Reset button is enabled when the current category has *either* known cards OR deferred cards (`canReset = knownCount > 0 || deferredCount > 0`) — disabled only when there's nothing to reset. The confirm prompt names the category. Reset clears both `knownIds` and `deferredIds` for that category.
12. **Progress UI is a thin progress bar + `N / total` text** in the header (user-chosen over alternatives like "remaining" count). The "total" is per-category; deferred cards count toward neither "known" nor "total-known" — they're still un-known.
13. **Direct-to-card on first load** — no intro screen. Default category is **Fish** (hardcoded in `loadSelectedCategory()` when localStorage is empty). The user picked this explicitly; don't auto-select "first non-empty category" — it's brittle and the user wants Fish even before other categories are added.
14. **Categories are independent decks.** `Category = 'Fish' | 'Algae' | 'Inverts' | 'UPC'`. The settings modal (hamburger top-left) swaps which deck is active. The selection persists across reloads via `localStorage` key `flashcards:v2:category`. On category change `App.tsx#handleCategoryChange` cancels any pending `flipResetTimer`, clears `lastAction` (undo doesn't survive category change — the user opted for this; simpler than scoping undo per-category), clears `isFlipped`, and nulls `currentId` so the deck-picker `useEffect` chooses a fresh card from the new category.
15. **Per-category "known" + "deferred" progress.** Both `knownIds` and `deferredIds` are independent across categories — the progress bar, `knownCount`, and `totalCards` are all scoped to the currently selected category (via `categoryCards = allCards.filter(c => c.category === selectedCategory)`). **Reset clears both `knownIds` and `deferredIds` for the current category**, not globally — the confirm prompt names the category. Implementation-wise this works because card IDs are category-prefixed (`fish/...` vs `algae/...`), so single flat `Set<string>`s in storage are enough.
16. **Empty categories show the existing `DeckEmpty` "no images yet" state.** Algae/Inverts/UPC all start empty and the modal lets the user pick them anyway — picking just swaps to a category with `totalCards === 0`, which triggers `DeckEmpty`. Don't disable or hide empty categories in the modal without asking.

## Concerns / things to watch for

- **`import.meta.glob` and asset URLs.** The chosen options (`eager: true, query: '?url', import: 'default'`) should give us strings to processed asset URLs. If glob returns `undefined` or unexpected shapes after a build, that's the first place to look. Files use case-insensitive extensions (`.JPG`, `.PNG`, etc. are also matched).
- **`images/` is at the project root**, not in `public/`. This is intentional — `import.meta.glob` doesn't see `public/`. If we ever move images to `public/`, the glob discovery breaks and we'd need a manual manifest.
- **localStorage on iOS Safari Private Mode** can throw on write. `storage.ts` swallows errors silently. The session will still work; nothing persists.
- **Card ID = `category/filename` means renames invalidate progress.** Renaming a file in `images/<cat>/` orphans its `localStorage` entry. **Moving a file between categories also invalidates progress** (different ID). Both are acceptable behavior — flag it if it becomes a real issue.
- **Hardcoded `<img>` `alt = card.name`** means the alt text reveals the answer to screen readers before the user "flips". For an a11y-focused user this would defeat the learning purpose. Out of scope for v1 but worth flagging if the user mentions accessibility.
- **No PWA / installable / offline support.** Easy to add with `vite-plugin-pwa` if user wants "save to home screen" with offline caching, but not requested.
