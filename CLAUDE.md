# Sea Life Flashcards — Context for Claude

## What this app is

A static, single-page web app that helps the user (Greg) memorize the names of sea creatures via flashcards. Each card has an image on one side and the species name on the other. The user reveals the name, then marks the card as **"I know it"** (removed from the rotation) or **"Practice"** (stays in the rotation). The deck is randomized. There is a one-step **Undo** for accidental taps and a **Reset** to put all cards back.

- **No login. No backend. No long-term server storage.**
- Per-user "known" state persists across page reloads via `localStorage` only.
- Mobile-first; desktop is just a centered narrower layout.
- Target deck size: ~100 cards (final set will be sea life; current `images/` has placeholder photos — `banana.jpg`, `greg.png`, `mona.jpg` — that the user will replace).

## Tech stack and why

- **React 18 + Vite + TypeScript** — fast dev, trivial static build, deployable to Vercel/Netlify/GitHub Pages.
- **Tailwind CSS v3** — mobile-first styling without writing component CSS.
- **No router, no state library.** Single-page, single-component-tree app. Plain `useState`/`useEffect` is sufficient and avoids unnecessary abstraction.
- **No test framework yet.** Manual testing is fine for this size; if logic grows (e.g., spaced-repetition weighting), add Vitest around `pickRandom` and the filename parser.

## Key design decisions (made deliberately — push back before changing)

1. **Two-stage image pipeline.**
   - **`images/`** is the user-managed source of truth. Raw photos, any of `.jpg/.jpeg/.png/.webp`, any case, any naming.
   - **`scripts/process-images.mjs`** (run via `npm run process-images`, also wired as `prebuild` so `npm run build` runs it automatically) reads `images/`, cleans each filename (strips embedded `jpg|jpeg|png|webp`, splits on `-_` and whitespace, title-cases each word, rejoins with spaces), compresses with `sharp` (resize: max 1200px on longest edge, JPEG quality 82, EXIF rotation honored), and writes to **`images-processed/`** as `.jpg`. The script is incremental (skips when dest is newer than source) and removes orphans from `images-processed/`.
   - **`cards.ts`** globs `/images-processed/*.jpg` via `import.meta.glob` — *not* the raw `images/` folder. Adding a photo to `images/` doesn't show up until you re-run `process-images` (or run `build`, which does it via the `prebuild` hook).
   - **Name collisions are fatal.** If two source files clean to the same target name, the script exits with an error pointing at both. Rename one of them.
   - At ~150 sea-life photos: raw `images/` was 31MB, processed `images-processed/` is 4.6MB (~85% reduction).
   - **Case-insensitive filesystems are handled** (macOS APFS default). The script reads `images-processed/` upfront and builds a lowercased lookup map. If a target's case differs from the on-disk name (e.g. user changed cleaner logic and `Oyt Foo.jpg` should now be `OYT Foo.jpg`), the script unlinks the old-cased file before sharp writes the new one — so renames take effect in a single pass. Output line `↻ recasing OLD → NEW` shows when this happens. Don't remove this dance; without it, sharp's write silently keeps the old casing on case-insensitive FS.
   - **Acronyms are preserved**: a word matching `/^[A-Z][A-Z0-9]+$/` (uppercase letter followed by 1+ uppercase letters or digits) keeps its case. So `OYT olive-yellowtail.jpg` → `OYT Olive Yellowtail.jpg`, and `KGBC kelp-gopher.jpg` → `KGBC Kelp Gopher.jpg`. Same logic exists in both `process-images.mjs` (`titleCaseWord`) and `cards.ts` (`titleCaseWord`) — keep them in sync if you change the rule.
2. **Filename → display name** (in `cards.ts`):
   - Strip extension.
   - Replace `-` and `_` with spaces (mostly a no-op now since `process-images` already converted them).
   - **Strip trailing digits** (`\s*\d+$`) — multiple photos of the same species in `images/` are named `Black Rockfish.jpg`, `Black Rockfish 2.jpg`, `Black Rockfish 3.jpg`. The disk filenames stay numbered (uniqueness), but the **display name** drops the suffix so all three read "Black Rockfish" on the card back. Each is still a separate card with its own ID, so marking one known doesn't hide the others (intentional — different photos test recognition robustness).
   - Title-case each word (mostly idempotent on processed input).
   - The parser still works on raw filenames if a developer ever points it back at `images/`.
   - No sidecar `names.json` override exists yet — if names get awkward (apostrophes, italics, scientific names), discuss with user before adding one.
3. **Card `id` is the filename string** (not a UUID). Stable across builds, used as the localStorage key entry. **If you change ID derivation, bump the storage key from `flashcards:v1:known` to `v2`** — old entries won't match and silently dropping known status is acceptable.
4. **Practice cards stay in the rotation with no weighting.** User explicitly said no boost/deprioritize. Don't add SRS-style spacing unless asked.
5. **Avoid same-card-twice-in-a-row** in `pickRandom.ts`. Just rerolls once if `avoidId` matches and there are >1 candidates. Don't over-engineer this.
6. **Image fit is `object-fit: contain`** (switched from `cover` because cropping was hiding parts of the fish — identifying features like fin shape and tail markings matter for ID). Letterbox space on the card front is plain white. Don't switch back to `cover` without asking.
7. **Card aspect ratio is `aspect-[8/5]`** (≈ 1.6 landscape) — chosen as the median aspect ratio across all 150 processed images. Verified with sharp metadata: 146 of 150 photos are landscape, median ratio 1.595, mean 1.661. A landscape card with `contain` minimizes letterboxing. Don't revert to portrait (`aspect-[3/4]` etc.) without re-running the stats.
7. **Flip animation is CSS 3D transforms**, ~700ms `rotateY`. Implemented with hand-rolled utility classes in `index.css` (`perspective-1000`, `preserve-3d`, `backface-hidden`, `rotate-y-180`) — no Tailwind plugin dependency.
8. **Undo is one step only** (last action). Stored in React state as `LastAction = { cardId, markedKnown } | null`. Cleared after use. `handleUndo` also clears any pending `flipResetTimer` so an in-flight card-swap can't fire after the undo restores state. Don't expand to a stack unless asked.
9. **Mid-flip card swap** (in `advanceToNext`): when the user marks a card, `setIsFlipped(false)` runs immediately, and the new card's `currentId` is set on a `setTimeout(..., 350ms)` — roughly halfway through the 700ms flip transition, when the back face has rotated past 90° and `backface-visibility: hidden` has kicked in. This is critical: if the swap happens synchronously, the user briefly sees the **next card's name** on the still-rotated back face before the unflip animates. Don't change the swap to fire synchronously; if you tune the flip duration, also retune `CARD_SWAP_DELAY_MS` to ~50% of it.
9. **Reset has a `window.confirm` prompt**, and the Reset button is disabled when `knownCount === 0`.
10. **Progress UI is a thin progress bar + `N / total` text** in the header (user-chosen over alternatives like "remaining" count).
11. **Direct-to-card on first load** — no intro screen.

Consider switching to `loading="lazy"` and only preloading the next card if perf is bad. Also worth recommending to the user that they keep images ≤300KB and use `.webp`.

## Concerns / things to watch for

- **`import.meta.glob` and asset URLs.** The chosen options (`eager: true, query: '?url', import: 'default'`) should give us strings to processed asset URLs. If glob returns `undefined` or unexpected shapes after a build, that's the first place to look. Files use case-insensitive extensions (`.JPG`, `.PNG`, etc. are also matched).
- **`images/` is at the project root**, not in `public/`. This is intentional — `import.meta.glob` doesn't see `public/`. If we ever move images to `public/`, the glob discovery breaks and we'd need a manual manifest.
- **localStorage on iOS Safari Private Mode** can throw on write. `storage.ts` swallows errors silently. The session will still work; nothing persists.
- **Card ID = filename means renames invalidate progress.** If the user renames `tuna.jpg` → `bluefin-tuna.jpg`, the old `tuna.jpg` entry in localStorage is orphaned and the renamed file is treated as new. This is acceptable behavior — flag it if it becomes a real issue.
- **Hardcoded `<img>` `alt = card.name`** means the alt text reveals the answer to screen readers before the user "flips". For an a11y-focused user this would defeat the learning purpose. Out of scope for v1 but worth flagging if the user mentions accessibility.
- **No PWA / installable / offline support.** Easy to add with `vite-plugin-pwa` if user wants "save to home screen" with offline caching, but not requested.
